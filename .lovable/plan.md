## Проблема

Модуль 1 (`draft-okr`) и Модуль 2 (`validate-okr`) сегодня оценивают OKR **по разным критериям**, поэтому одна и та же формулировка получает «зелёный» в Модуле 1 и «красный» в Модуле 2.

Что именно расходится:

| Аспект | Модуль 1 (draft-okr) | Модуль 2 (validate-okr) |
|---|---|---|
| Источник оценки | `score_hint` — свободная самооценка LLM | Жёсткий чек-лист из 8 правил (O1–O3, KR1–KR4, KR10) |
| Правило O3 (нет цифр в Objective) | Не упомянуто | Critical |
| Правило KR2 (явные baseline/target) | «если неизвестно — оставь пустым + warning» (т.е. валидно с пустыми) | Critical-замечание, если пусто |
| Правило KR10 (есть leading) | «если возможно» | Important |
| Запрещённые activity-глаголы | Свой список | Свой пересекающийся, но не идентичный список |
| Дополнительно | `horizon_fit` (свой блок, которого нет в M2) | — |

Итог: Модуль 1 ставит 75–85, потому что считает «horizon_fit ок + 2–3 KR есть = хорошо», а Модуль 2 валит на O3/KR2/KR10.

## Цель

Сделать так, чтобы зелёная оценка в Модуле 1 **гарантировала** прохождение Модуля 2 (или прозрачно объясняла, чего не хватает). Без редизайна, без новой архитектуры, без слияния модулей — Модуль 2 остаётся главным аудитором.

## Что предлагаю (3 шага, без переделки UX)

### Шаг 1. Единый источник правил (shared rules block)

Создать `supabase/functions/_shared/okr_rules.ts` с **одним** блоком текста — каноничным описанием правил O1–O3, KR1–KR4, KR10 (severity + формулировки). Импортировать его и в `draft-okr`, и в `validate-okr`. Сейчас правила задублированы в обоих system-промптах с разными формулировками — это и есть корень рассинхрона.

Влияние на интерфейсы: **нет**. Меняются только тексты системных промптов на сервере.

### Шаг 2. Усилить самооценку Модуля 1 правилами Модуля 2

В `draft-okr`:

- В system prompt вставить тот же rules-блок и явно потребовать: «`score_hint` рассчитывается как доля пройденных правил с весами critical=3, important=2, improve=1; если есть хотя бы одно проваленное `critical` — `score_hint ≤ 60`».
- Добавить в JSON-схему ответа опциональное поле `self_audit: { critical_fails: string[], important_fails: string[] }` — короткий список ID проваленных правил по версии самого драфтера.
- В `OkrGenerator.tsx` (Модуль 1) показать эти ID маленьким бейджем рядом с «Самооценка X/100» — пользователь сразу видит «есть 1 critical-замечание» и не удивляется красному в Модуле 2.

Поведение «передать в аудит» **не меняется** — кнопка остаётся, Модуль 2 всё равно прогоняет полную проверку.

### Шаг 3. (Опционально, по согласованию) Smart-mode «Сразу с аудитом»

Добавить в Модуле 1 чекбокс **«Сразу прогнать через аудит»** (по умолчанию выкл.). Если включён — после `draft-okr` фронт автоматически дёргает `validate-okr` и подставляет результат в правую панель. Это закрывает кейс «я думал что готово, а оно не готово» одним кликом.

Без чекбокса — текущий workflow сохраняется на 100%.

## Что НЕ трогаю

- Модуль 3 (Solutions / `validate-solution`) — без изменений.
- Архитектуру передачи стейта `Index.tsx` (`handleGenerated`, `handleSendToSolutions`).
- Визуальный стиль, layout, дизайн-токены.
- Список моделей, edge-functions для AI gateway.
- Severity / «Почему важно» в Модуле 2 — только что добавили, оставляем.
